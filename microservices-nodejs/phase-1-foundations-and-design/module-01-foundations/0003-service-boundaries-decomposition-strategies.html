Service Boundaries & Decomposition Strategies

How to safely rip apart a monolithic Node.js application without causing a production outage. Say goodbye to massive downtime and hello to the Strangler Fig.

⏱ 35 min read Phase 1 — Foundations & Design Prereq: Lesson 2

You can't do it all at once

The biggest mistake companies make when migrating to microservices is attempting a "Big Bang" rewrite. They pause feature development for 6 months, rewrite the entire monolith into 20 microservices, and try to flip the switch in one night. It almost always results in disaster.

Instead, we use incremental decomposition strategies. You extract one Bounded Context at a time, verify it works in production, and then move on to the next. The system is always running.

1. The Strangler Fig Pattern

The Strangler Fig Pattern involves placing an API Gateway or Proxy in front of your monolith. Initially, 100% of traffic routes to the monolith. When you extract a service (like Campaign Management), you configure the proxy to route /v1/campaigns/* to the new service, while everything else falls back to the monolith.

In Node.js, you can build this easily using Express and http-proxy-middleware:

// api-gateway.js
const express = require('express');
const { createProxyMiddleware } = require('http-proxy-middleware');

const app = express();

// 1. Route specific paths to the new microservice
app.use('/v1/campaigns', createProxyMiddleware({ 
  target: 'http://campaign-service:3001', 
  changeOrigin: true 
}));

// 2. Everything else falls back to the old monolith
app.use('/', createProxyMiddleware({ 
  target: 'http://legacy-monolith:3000', 
  changeOrigin: true 
}));

app.listen(8080);

2. Branch by Abstraction

Sometimes, logic is deeply embedded inside the monolith (like your `helper.js`) and isn't exposed via a clean REST endpoint. In this case, you use Branch by Abstraction:

  1. Create an Interface/Abstraction over the legacy code.
  2. Change all clients in the monolith to call the Abstraction instead of the legacy code directly.
  3. Build the new Microservice.
  4. Create a new implementation of the Abstraction that makes an HTTP call to the new Microservice.
  5. Flip the switch (via feature flag) so the Abstraction calls the microservice instead of the legacy code.
  6. Once stable, delete the legacy code.

3. Parallel Run (Dark Launching)

For highly critical paths (like Payments or Core Campaigns), you can't afford a single bug. With a Parallel Run, you execute BOTH the old code and the new microservice simultaneously. You return the monolith's result to the user, but you compare it against the new service's result in the background.

Failure Path Warning

If the new microservice is slow or crashes during a Parallel Run, your monolith must catch the error and gracefully ignore it so the user's request isn't impacted. Make sure to wrap the call to the new service in a `try/catch` and a timeout.

Strategy Tradeoffs

Strategy Best For Pros Cons
Strangler Fig Extracting entire REST resource routes (e.g., `/users`) Clean boundary at the network level; zero downtime. Requires an API Gateway; hard if DB is highly coupled.
Branch by Abstraction Extracting internal shared libraries or core domain logic Safe, iterative, allows easy rollback via feature flags. Requires touching lots of legacy code to introduce the abstraction.
Parallel Run High-risk/financial systems where bugs cost money Proves 100% parity before going live. Double the compute cost; complex to compare state-changing operations (like DB writes).

🧠 Retrieval Practice

1. What is the Strangler Fig pattern in the context of microservices?
Reveal answer

A migration strategy where you place an API Gateway in front of your legacy monolith. Over time, you route specific endpoints to new microservices while the rest falls back to the monolith.

2. When would you use Branch by Abstraction instead of the Strangler Fig pattern?
Reveal answer

When the code you want to extract is deeply embedded internal logic (like a shared library or `helper.js`) that isn't exposed via a clean REST endpoint.
