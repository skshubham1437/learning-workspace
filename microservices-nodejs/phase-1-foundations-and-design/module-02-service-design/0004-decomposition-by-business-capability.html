Decomposition by Business Capability

The right way to split a monolith is not by technical layer (routes, controllers, models) but by what the business does. This lesson teaches the tactical DDD patterns that structure the code inside each service.

⏱ 40 min read Phase 1 — Foundations & Design Prereq: Lesson 3

📖 Primary source: Chris Richardson, "Decompose by Business Capability" — the canonical pattern reference. Read alongside Vaughn Vernon, Implementing Domain-Driven Design Ch. 10 ("Aggregates") and Sam Newman, Building Microservices Ch. 3 ("Splitting the Monolith").

The two wrong ways to decompose

In Module 01, we said "don't map folders to services." But there's actually two bad decomposition strategies, and both are tempting because they match the way your monolith is already organised.

Strategy What it looks like Why it fails
By technical layer "API Gateway Service," "Business Logic Service," "Data Service" — splitting your routes/, service/, and models/ into separate deployables Every user request now crosses 3 network boundaries instead of 0. You've added latency and failure modes without gaining any independence. All three services still change together for every feature.[1]
By entity/table "User Service," "Campaign Service," "Payment Service" — one service per database table Tables don't map to business capabilities. A "campaign" involves creators, payments, deliverables, and notifications. Splitting by table means every business operation spans 4+ services — you've built a distributed monolith.

The right way: decompose by business capability

A business capability is something the organisation does to generate value.[2] Not a database table, not a code module — a real-world activity.

Applied to your monolith

Your routes/v1/index.js mounts 30+ route modules. But the business doesn't have 30 capabilities. It has roughly these:

  • Campaign Execution — creating, managing, and completing brand–creator campaigns
  • Creator Discovery & Identity — onboarding creators, managing profiles, connecting social accounts
  • Financial Operations — payouts, KYC verification, coupons, affiliate earnings
  • Brand Engagement — brand onboarding, service packages, inbound requests
  • Communication — email, WhatsApp, push notifications (all channels, one capability)
  • Intelligence — analytics, reports, AI features

Each business capability becomes a candidate service boundary. The test is simple: can a team own this capability end-to-end and ship changes without coordinating with other teams? If yes, it's a good boundary. If no, your boundary is in the wrong place.

What goes inside a service — Tactical DDD

In Module 01, we used Strategic DDD to decide where to draw the boundaries. Now we zoom in. Tactical DDD tells you how to structure the code inside each service.[3]

There are four core building blocks:

Building Block What it is Example from your system
Entity An object with a unique identity that persists over time. Two entities with the same data but different IDs are different things. A Campaign — it has an ID, a lifecycle, and its identity matters even if the title changes.
Value Object An object defined by its attributes, not identity. Two value objects with the same data are interchangeable. A Money({ amount: 5000, currency: 'INR' }) — you don't care which ₹5,000 it is. If two have the same values, they're equal.
Aggregate A cluster of Entities and Value Objects treated as a single unit for data changes. Has one Aggregate Root that controls all access. A Campaign aggregate containing its Deliverables, CreatorAssignments, and Budget — you never modify a deliverable directly, you go through the Campaign.
Repository An abstraction that provides access to Aggregates as if they were in-memory collections. Hides the database. A CampaignRepository with methods like findById(), save() — the domain code never sees SQL or MongoDB queries.
The most important one: the Aggregate

The Aggregate is the single most important tactical pattern for microservices. It defines the consistency boundary: everything inside an Aggregate is always consistent (strong consistency), but between Aggregates, you accept eventual consistency.[3]

This is why Aggregates matter for microservices: one service should own one or more Aggregates, and cross-Aggregate communication happens through events or API calls — never direct database access.

What your monolith does instead

Your campaign.js is 19,000+ lines. That's not just big — it's a sign that there are no Aggregate boundaries inside the Campaign module. Everything touches everything.

❌ What a 19,000-line service file looks like

// campaign.js — a god file
// Line 1-2000: Campaign CRUD
// Line 2001-5000: Creator assignment logic
// Line 5001-8000: Deliverable tracking
// Line 8001-11000: Payment calculation
// Line 11001-14000: Notification triggers
// Line 14001-17000: Analytics and reporting
// Line 17001-19600: Music campaign variants

// The problem: a change to payment logic at line 9000
// might silently break deliverable tracking at line 6500
// because they share local variables and helper functions.

This happens because the monolith has no concept of Aggregates. Any function can reach into any data. There's no transactional boundary — everything shares one giant request-level transaction.

✅ What the same domain looks like with Aggregates

// campaign-service/
//   domain/
//     campaign/              ← Aggregate Root
//       Campaign.js           (Entity: identity, lifecycle, status)
//       Deliverable.js        (Entity: belongs to Campaign)
//       Budget.js             (Value Object: amount + currency)
//       CampaignCreated.js    (Domain Event)
//     creator-assignment/     ← Separate Aggregate
//       Assignment.js         (Entity: links creator to campaign)
//       Rate.js               (Value Object: per-post rate)
//   repository/
//     CampaignRepository.js  (persistence abstraction)
//   application/
//     CreateCampaign.js      (use case / application service)
//     AssignCreator.js       (use case)

Notice: Deliverable is inside the Campaign Aggregate because a deliverable cannot exist without a campaign. But Assignment is a separate Aggregate — a creator assignment has its own lifecycle and can be modified independently.

Entity vs Value Object — the practical test

This distinction trips people up. Here's the one question that settles it:[3]

The identity test

If two objects have the same data, are they the same thing?

  • Yes → Value Object. Two Money({ amount: 5000, currency: 'INR' }) instances are interchangeable. You'd never say "give me that specific ₹5,000."
  • No → Entity. Two Campaign objects with the same title and budget are still different campaigns. They have different IDs, different histories, different creators assigned.

In code

// Value Object — immutable, compared by value
class Money {
  constructor(amount, currency) {
    if (amount < 0) throw new Error('Amount cannot be negative');
    if (!currency) throw new Error('Currency is required');

    // Freeze to enforce immutability
    this.amount = amount;
    this.currency = currency;
    Object.freeze(this);
  }

  equals(other) {
    return other instanceof Money
      && this.amount === other.amount
      && this.currency === other.currency;
  }

  add(other) {
    if (this.currency !== other.currency)
      throw new Error('Cannot add different currencies');
    return new Money(this.amount + other.amount, this.currency);
  }
}

// Entity — has identity, mutable state
class Campaign {
  #id; #title; #budget; #status; #deliverables;

  constructor({ id, title, budget, status = 'draft' }) {
    this.#id = id;
    this.#title = title;
    this.#budget = budget;  // a Money value object
    this.#status = status;
    this.#deliverables = [];
  }

  get id() { return this.#id; }

  addDeliverable(deliverable) {
    if (this.#status !== 'draft')
      throw new Error('Cannot add deliverables to a non-draft campaign');
    this.#deliverables.push(deliverable);
  }

  launch() {
    if (this.#deliverables.length === 0)
      throw new Error('Cannot launch a campaign without deliverables');
    this.#status = 'active';
    // Returns a domain event (we'll cover events in Module 03)
    return { type: 'CampaignLaunched', campaignId: this.#id };
  }
}
Why this matters for your monolith

Right now, your campaign code likely has bare objects like { amount: 5000, currency: 'INR' } scattered across 20 functions. There's no validation at the domain level — it's all done in route handlers or (worse) not at all. When you extract a Campaign Service, these Value Objects become the first line of defense against bad data entering the system.

The Repository Pattern — hiding the database

In your monolith, service functions probably call the database directly: 

// ❌ What you probably have — database coupled to business logic
async function getCampaign(campaignId) {
  const campaign = await db.collection('campaigns').findOne({ _id: campaignId });
  const deliverables = await db.collection('deliverables').find({ campaignId }).toArray();
  return { ...campaign, deliverables };
}

The Repository pattern creates an abstraction between your domain logic and the database. Your domain code thinks it's working with an in-memory collection:[3] 

// ✅ Repository — domain code doesn't know about MongoDB
class CampaignRepository {
  #collection;
  #deliverableCollection;

  constructor(db) {
    this.#collection = db.collection('campaigns');
    this.#deliverableCollection = db.collection('deliverables');
  }

  async findById(campaignId) {
    const raw = await this.#collection.findOne({ _id: campaignId });
    if (!raw) return null;

    const deliverables = await this.#deliverableCollection
      .find({ campaignId })
      .toArray();

    // Reconstruct the Aggregate from persistence
    return new Campaign({
      id: raw._id,
      title: raw.title,
      budget: new Money(raw.budgetAmount, raw.budgetCurrency),
      status: raw.status,
      deliverables: deliverables.map(d => new Deliverable(d)),
    });
  }

  async save(campaign) {
    // Persist the entire Aggregate atomically
    const session = await this.#collection.client.startSession();
    try {
      await session.withTransaction(async () => {
        await this.#collection.replaceOne(
          { _id: campaign.id },
          campaign.toPersistence(),
          { upsert: true, session }
        );
        // ... save deliverables in same transaction
      });
    } finally {
      await session.endSession();
    }
  }
}
Why bother with this abstraction?

When you eventually migrate from MongoDB to PostgreSQL (or split the database), only the Repository changes. The domain code — all your business rules — stays untouched. This is the same principle as the Anti-Corruption Layer, but applied inside your own service between the domain and persistence layers.

Four rules for designing Aggregates

Vernon's four rules[3] keep Aggregates small and independently deployable:

Rule What it means Example
1. Protect invariants inside the Aggregate All business rules that must be enforced atomically live inside one Aggregate "A campaign cannot launch without at least one deliverable" — checked inside Campaign.launch()
2. Design small Aggregates Don't put the entire domain into one god Aggregate. Each Aggregate should be a small consistency boundary. Campaign and CreatorAssignment are separate Aggregates — assignment changes shouldn't lock the campaign.
3. Reference other Aggregates by ID only Don't hold a direct reference to another Aggregate instance. Just store its ID. An Assignment holds campaignId and creatorId, not the full Campaign or Creator objects.
4. Use eventual consistency across Aggregates Don't try to update two Aggregates in one database transaction. Use domain events instead. When a Campaign is completed, a CampaignCompleted event triggers a payout in the Finance Aggregate.
The #1 mistake — the God Aggregate

Your 19,000-line campaign.js is essentially a God Aggregate. Everything related to campaigns — assignments, deliverables, payments, notifications — lives in one transactional blob. This is why a small change to payment calculation can break deliverable tracking: they share the same transaction boundary with no isolation.

Splitting this into smaller Aggregates is the first step of your microservices migration — and you can do it inside the monolith, before extracting any service.

🧠 Retrieval Practice

Answer from memory before revealing. The slight difficulty is the point — it's where storage strength builds.

1. Name the two wrong decomposition strategies and explain why each fails. RECALL-1D
Reveal answer

By technical layer (routes/controllers/models as separate services) — adds network hops without independence; all layers still change together.
By entity/table (one service per DB table) — business operations span multiple tables, so every request fans out across services; creates a distributed monolith.

2. What is the practical test to distinguish an Entity from a Value Object? RECALL-1D
Reveal answer

"If two objects have the same data, are they the same thing?" Yes → Value Object (compared by value, immutable). No → Entity (has unique identity, mutable over its lifecycle).

3. What is an Aggregate, and why is it the most important tactical DDD pattern for microservices? RECALL-3D
Reveal answer

An Aggregate is a cluster of Entities and Value Objects treated as a single transactional unit, accessed through an Aggregate Root. It defines the consistency boundary: strong consistency inside, eventual consistency outside. This maps directly to microservice boundaries — one service owns one or more Aggregates.

4. Name all four Aggregate design rules from Vernon. RECALL-3D
Reveal answer

1. Protect invariants inside the Aggregate.
2. Design small Aggregates.
3. Reference other Aggregates by ID only (not by direct object reference).
4. Use eventual consistency across Aggregates (domain events, not shared transactions).

5. What does the Repository pattern do, and why is it critical during a database migration? RECALL-7D
Reveal answer

The Repository creates an abstraction between domain logic and the database, making Aggregates appear as in-memory collections. During a DB migration (e.g., MongoDB → PostgreSQL), only the Repository implementation changes — all domain and business logic stays untouched.

Model the Campaign Aggregate

Create a folder playground/campaign-aggregate/ and build the following:

  1. A Money Value Object — immutable, with validation (no negative amounts, currency required), and an equals() method.
  2. A Campaign Entity (Aggregate Root) — with a launch() method that enforces the invariant: "cannot launch without at least one deliverable." The launch() method should return a domain event object { type: 'CampaignLaunched', campaignId }.
  3. A simple CampaignRepository — for now, use an in-memory Map instead of a real database. It should have save(campaign) and findById(id).
  4. A test script that:
    • Creates a campaign, adds a deliverable, launches it, and verifies the event is returned.
    • Tries to launch a campaign with zero deliverables and catches the error.
    • Saves and retrieves a campaign from the in-memory repository.

Success criterion: Your Campaign.launch() throws if the invariant is violated, and your Repository can round-trip a Campaign aggregate.
