# Mission: Microservices & Distributed Systems with Node.js

## Why

Transition from a production Node.js monolith engineer to a distributed systems architect who can design, build, migrate, and operate microservices systems — and defend those decisions confidently in Staff-level system design interviews.

## Success looks like

- Design microservice architectures from scratch with justified tradeoffs
- Migrate a production Express.js monolith into microservices using proven patterns (Strangler Fig, etc.)
- Build production-ready Node.js microservices with proper error handling, observability, and resilience
- Implement all major communication patterns (sync: REST/gRPC/GraphQL, async: Kafka/RabbitMQ/Redis Streams)
- Handle distributed system failure modes that don't exist in monoliths (network partitions, cascading failures, eventual consistency)
- Deploy and operate microservices on Kubernetes
- Discuss all architectural tradeoffs clearly in Staff and System Design interviews
- Build a complete e-commerce platform from monolith to microservices as a capstone project

## Constraints

- Learning pace: ~1 hour/day
- Existing stack: Node.js, Express, MySQL, MongoDB, Docker — bridge from these, don't re-teach them
- No prior experience with: Kafka, RabbitMQ, gRPC, Istio, Kubernetes in production, service meshes, distributed transactions
- Preference: Production-quality code and real-world examples, not tutorial-quality toy projects

## Out of scope

- Re-learning Node.js internals, Express middleware, Docker basics, SQL/NoSQL fundamentals, auth patterns
- Frontend/UI concerns
- Non-Node.js runtimes (Go, Java, Rust microservices) unless for comparison context
