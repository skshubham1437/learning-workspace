Lesson 04

Your First CI Pipeline

Write a real workflow that lints and tests a Node.js app — and learn why needs: is not uses:.

📖 Primary source: Workflow Syntax Reference — the single most important page in the GitHub Actions docs. Bookmark it.

One Thing to Fix First

After Lesson 3, I asked how to make one job wait for another. You said uses: lint. The instinct was exactly right — job B should declare a dependency on job A. But the keyword is needs:, not uses:.

These two keywords live at different levels and mean completely different things:

KeywordLives on a…Means
uses:Step"Load this pre-built Action" (a plugin)
needs:Job"Wait for this other job to finish"
Mnemonic

A step uses a tool. A job needs another job to go first.

Vocabulary Check

We keep saying "pipeline" and "workflow." Quick distinction: a pipeline is the abstract idea — the sequence of stages code passes through. A workflow is GitHub Actions' concrete implementation of that idea — a YAML file in .github/workflows/. When someone says "write a CI pipeline," they mean "write a workflow file."

The App

We need a small, real Node.js project to run our pipeline against. Here's the minimal structure:

my-node-app/
├── .github/
│   └── workflows/
│       └── ci.yml          # our workflow
├── src/
│   └── math.js             # a tiny module
├── tests/
│   └── math.test.js        # a test for it
├── package.json
└── .eslintrc.json

src/math.js

function add(a, b) {
  return a + b;
}

function multiply(a, b) {
  return a * b;
}

module.exports = { add, multiply };

tests/math.test.js

const { describe, it } = require('node:test');
const assert = require('node:assert');
const { add, multiply } = require('../src/math');

describe('math', () => {
  it('adds two numbers', () => {
    assert.strictEqual(add(2, 3), 5);
  });

  it('multiplies two numbers', () => {
    assert.strictEqual(multiply(4, 5), 20);
  });
});

We're using Node's built-in test runner (available since Node 20) — no Jest or Mocha needed.

package.json

{
  "name": "my-node-app",
  "scripts": {
    "test": "node --test tests/",
    "lint": "eslint src/"
  },
  "devDependencies": {
    "eslint": "^8.56.0"
  }
}

.eslintrc.json

{
  "env": { "node": true, "es2022": true },
  "rules": {
    "no-unused-vars": "error",
    "no-undef": "error"
  }
}

The Workflow

Now the main event. This workflow runs on every push to main and on every pull request targeting main. It has two jobs: lint first, then test.

name: "CI"

on:
  push:
    branches: [ "main" ]
  pull_request:
    branches: [ "main" ]

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'

      - name: Install dependencies
        run: npm ci

      - name: Run linter
        run: npm run lint

  test:
    needs: lint              # ← waits for 'lint' to pass
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'

      - name: Install dependencies
        run: npm ci

      - name: Run tests
        run: npm test

Notice the two triggers under on:. Running on push validates code after it lands on main. Running on pull_request validates code before it merges — acting as a gate. Together, they form a safety net.

Why npm ci?

You'll notice we use npm ci instead of npm install. In CI, you want a clean, deterministic install from your lockfile (package-lock.json). npm ci does exactly that — it deletes node_modules/, installs only what the lockfile says, and never modifies the lockfile. It's faster and more predictable. Always prefer npm ci in pipelines.

The Gotcha: Job Isolation

Look at the workflow above. Both lint and test do the exact same setup: checkout → setup Node → install dependencies. That's not a mistake — it's a consequence of a fundamental rule:

Crucial Concept

Every job gets its own fresh, isolated runner. The needs: keyword controls ordering, not sharing. Job B waits for job A to succeed — but job B still starts on a brand new, empty machine.

Analogy

Imagine two kitchens in different buildings. You can tell Kitchen B to wait until Kitchen A finishes — but you still have to send the ingredients and recipe to both kitchens separately.

This means if a job needs your code, it must check it out. Every. Single. Time.

(There are ways to share files between jobs using artifacts, but that's an advanced topic for later.)

🔍 Spot the Bugs

This workflow has two bugs. Try to find them before revealing the answers.

name: "CI"

on:
  push:
    branches: [ "main" ]

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm run lint

  test:
    uses: lint
    runs-on: ubuntu-latest
    steps:
      - name: Run tests
        run: npm test

Bug #1

Look at the test job. What's wrong with how it declares its dependency on lint?
Reveal answer

It says uses: lint, but uses: is a step-level keyword for loading Actions. The correct keyword is needs: lint. GitHub would either error out or try to interpret lint as an Action reference — neither is what you want.

Bug #2

Even after fixing Bug #1, the test job would still fail. Why?
Reveal answer

The test job has no actions/checkout step and no Node.js setup. It jumps straight to npm test on a fresh, empty runner — there's no code, no Node.js, and no npm. Remember: each job is isolated and must set up its environment from scratch.

🧠 Retrieval Practice

1. What keyword makes one job wait for another to complete?
Reveal answer

needs: — placed at the job level (not inside steps:). Example: needs: lint makes the current job wait for the lint job to succeed.

2. Job A runs npm ci and installs all dependencies. Job B has needs: A. Does job B already have the dependencies installed?
Reveal answer

No. needs: controls ordering, not sharing. Job B gets its own fresh runner and must install everything itself.

3. What's the difference between npm ci and npm install in a CI context?
Reveal answer

npm ci does a clean, deterministic install from the lockfile. It's faster, won't update package-lock.json, and is what you should always use in pipelines. npm install may modify the lockfile.

4. Our workflow triggers on both push and pull_request. Why both?
Reveal answer

push runs CI after code lands on main — catching anything that slipped through. pull_request runs CI before merging — giving you a gate to reject bad code. Together, they form a safety net.

Your Turn

The best way to lock this in is to build it. Here's what to do:

  1. Create a new GitHub repository (or use an existing one).
  2. Add the files shown above: src/math.js, tests/math.test.js, package.json, and .eslintrc.json.
  3. Run npm install locally to generate package-lock.json (CI needs this for npm ci).
  4. Create .github/workflows/ci.yml with the workflow we wrote.
  5. Push to main and open the Actions tab in GitHub.
  6. Watch both jobs run: lint first, then test.
