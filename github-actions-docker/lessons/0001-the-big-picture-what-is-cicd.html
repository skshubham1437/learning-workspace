Lesson 01

The Big Picture: What Is CI/CD?

Before touching any YAML or Docker commands, let's build the mental model that makes everything else click.

The Problem CI/CD Solves

Imagine you're working on a team. You write code on your laptop. Your teammate writes code on theirs. At some point you both need to merge your changes, check that nothing broke, and get the result in front of users.

Without automation, this process looks like:

  1. Merge code manually and hope for the best
  2. Run tests locally — if you remember to
  3. Build the app on your machine — "works on my machine" 🤷
  4. Copy files to a server via FTP or SSH
  5. Pray nothing is different about the server's environment

Every single one of these steps is a place where things can silently go wrong. CI/CD automates them so they can't be forgotten, skipped, or done inconsistently.

Two Separate Ideas

The term "CI/CD" glues two distinct concepts together. Let's pull them apart:

CI — Continuous Integration

"Does the code work?"

Every time someone pushes code, an automated system:

  • Pulls the latest code
  • Installs dependencies
  • Runs linters (style checks)
  • Runs tests
  • Builds the application

If any step fails, the team knows immediately — before the code reaches production. The key word is continuous: it happens on every change, not once a week.

CD — Continuous Delivery / Deployment

"Get the working code to users."

Once CI proves the code works, CD takes the built artifact and moves it toward production:

  • Continuous Delivery — the code is ready to deploy at any time (a human presses the button)
  • Continuous Deployment — the code is automatically deployed (no human needed)
Key distinction

CI answers "is it safe?" — CD answers "how does it reach users?" They're different concerns, and many teams start with CI only. That's perfectly fine.

The Pipeline

A CI/CD pipeline is the sequence of automated steps your code passes through. Here's what a typical pipeline looks like:

← Continuous Integration →
📥 Checkout
→
📦 Install
→
🔍 Lint
→
✅ Test
→
🔨 Build
← Continuous Delivery →
🐳 Package
→
📤 Push
→
🚀 Deploy

Notice how the left side (CI) is about validating and the right side (CD) is about delivering. The pipeline only moves right if the left side passes.

Where Docker Fits In

Docker solves the "works on my machine" problem. Instead of deploying raw files and hoping the server has the right version of Node, Go, or nginx installed, you package your entire application — code, dependencies, runtime — into a container image.

Analogy

Think of a container image like a shipping container (that's actually where the name comes from). It doesn't matter what's inside — the crane, the truck, and the ship all handle it the same way. Similarly, a Docker container runs the same way on your laptop, in CI, and in production.

In a CI/CD pipeline, Docker typically appears in two places:

  1. As the build environment — CI runs your tests inside a container so the environment is guaranteed identical every time
  2. As the deployment artifact — instead of deploying files, you build a Docker image and push it to a registry (like Docker Hub or GitHub Container Registry), then the server pulls and runs that image

Where GitHub Actions Fits In

GitHub Actions is the orchestrator — it's the engine that runs your pipeline. When you push code to GitHub, Actions reads a YAML file in your repository and executes the steps you defined.

The key concepts (we'll go deep on these in future lessons):

  • Workflow — a YAML file that defines your pipeline (lives in .github/workflows/)
  • Trigger — what starts the workflow (e.g., a push, a pull request, a schedule)
  • Job — a group of steps that run on the same machine
  • Step — a single command or action within a job
  • Runner — the machine (virtual, provided by GitHub) where your job executes
Mental model

GitHub Actions = the automation engine.
Docker = the packaging format.
Your code = the thing being automated and packaged.
They're three layers of the same system, not competing alternatives.

How This Applies to Your Three Stacks

The pipeline pattern is the same for every stack — only the build commands and Dockerfile instructions change:

Stack CI: Test CI: Build Docker: Final image
React npm test npm run build → static files nginx serving /dist
Node/Express npm test No compile step (interpreted) node running server.js
Go go test ./... go build → single binary Tiny alpine running the binary

Notice the pattern: the pipeline shape is identical. Only the commands inside each stage change. This is why understanding the model matters more than memorising YAML — once you get the model, every stack is just filling in blanks.

What you now know

🧠 Retrieval Practice

Try to answer these from memory before revealing the answers. This is where real learning happens — it should feel slightly effortful.

1. What is the difference between Continuous Delivery and Continuous Deployment?
Reveal answer

Continuous Delivery means the code is always ready to deploy — a human makes the final call. Continuous Deployment means it goes out automatically with no human gate. Delivery is the safer starting point.

2. In a CI/CD pipeline, where does Docker typically appear? (Name two places.)
Reveal answer

(1) As the build/test environment — CI runs inside a container for consistency. (2) As the deployment artifact — you build a Docker image and push it to a registry, then the server pulls and runs it.

3. A React app, a Node/Express app, and a Go app all go through the same CI/CD pipeline. What stays the same and what changes?
Reveal answer

The pipeline shape stays the same (checkout → install → lint → test → build → package → push → deploy). The commands inside each stage change (e.g., npm test vs. go test, nginx image vs. alpine image).

4. What problem does Docker solve that raw file deployment doesn't?
Reveal answer

Docker eliminates "works on my machine" by packaging the application and its entire environment (runtime, dependencies, OS libraries) into a single portable image. The same image runs identically everywhere.

5. Name the five key GitHub Actions concepts (the hierarchy from top to bottom).
Reveal answer

Workflow (the YAML file) → triggered by a Trigger (push, PR) → contains Jobs (groups of steps on one machine) → each job has Steps (commands/actions) → each job runs on a Runner (GitHub's virtual machine).
