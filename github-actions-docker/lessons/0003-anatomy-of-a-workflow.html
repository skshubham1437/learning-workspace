Lesson 03

Anatomy of a GitHub Actions Workflow

Triggers, jobs, steps, and runners: translating the CI/CD pipeline into YAML.

The Big Picture (Recap)

In Phase 1, we learned that GitHub Actions is the orchestrator. It's the engine that listens for events (like a code push) and executes your pipeline (linting, testing, building).

We tell GitHub Actions what to do by writing a Workflow file in YAML format. It must be placed in a very specific folder: .github/workflows/ at the root of your repository.

The Hierarchy

Every workflow is built from a strict hierarchy of four concepts:

1. Workflow (The Pipeline)
A single YAML file triggered by an event (e.g., "Push to main branch").
2. Job (The Environment)
A grouping of steps that execute on the same Runner (a virtual machine, like Ubuntu). Jobs run in parallel by default.
3. Step (The Action)
Runs a pre-built Action (e.g., "Checkout code").
3. Step (The Action)
Runs a shell command (e.g., npm test).

Reading the YAML

Let's look at a real, bare-bones workflow that tests a Node.js app on every push to the main branch.

name: "Node.js CI"

# 1. THE TRIGGER
on:
  push:
    branches: [ "main" ]

# 2. THE JOBS
jobs:
  test-app:
    # 3. THE RUNNER
    runs-on: ubuntu-latest

    # 4. THE STEPS
    steps:
      # Step A: Get the code
      - name: Checkout repository
        uses: actions/checkout@v4

      # Step B: Install Node.js
      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'

      # Step C: Install dependencies
      - name: Install dependencies
        run: npm ci

      # Step D: Run tests
      - name: Run tests
        run: npm test

Let's break it down:

1. The Trigger (on:)
This tells GitHub when to run the workflow. Here, it runs whenever someone pushes to the main branch. You can also trigger on Pull Requests (pull_request:), on a schedule (like a cron job), or even manually (workflow_dispatch:).

2. The Jobs (jobs:)
We defined one job named test-app. If we had another job called deploy-app, GitHub would try to run them both at the same time (in parallel) unless we explicitly told it to wait.

3. The Runner (runs-on:)
ubuntu-latest tells GitHub: "Please spin up a fresh, empty Ubuntu Linux virtual machine to run my job." Every time this job runs, it gets a completely brand new, clean machine.

Crucial Concept

Because the runner is a fresh machine every time, your code isn't there yet. Neither is Node.js, Go, or your dependencies. You have to explicitly instruct the runner to download your code and set up the environment.

4. The Steps (steps:)
Steps execute sequentially on the runner. There are two types of steps:

  • uses: — This runs a pre-packaged "Action" written by the community or GitHub.
    Example: actions/checkout@v4 is a magic script that runs git clone and downloads your repo into the runner. You almost always need this as your first step!
  • run: — This simply executes a terminal command on the runner, exactly as if you typed it on your laptop.
    Example: npm ci (clean install) or go build.

Why `actions/checkout`?

This is the most common point of confusion for beginners. They write a step that says run: npm test and the workflow fails with "File not found."

Remember: the workflow is just a script running on a cloud server somewhere. That server knows nothing about your repository until you tell it to download it. actions/checkout bridges that gap.

🧠 Retrieval Practice

1. In which exact folder must you place your workflow YAML files for GitHub to detect them?
Reveal answer

.github/workflows/ at the root of your repository.

2. What happens if you put two jobs (e.g., `test` and `lint`) under the `jobs:` section? Do they run sequentially or in parallel by default?
Reveal answer

They run in parallel by default. Each job gets its own separate runner machine.

3. What is the difference between `uses:` and `run:` in a step?
Reveal answer

uses: executes a pre-built Action from the community (like a plugin), while run: executes a raw shell command (like `npm test`) on the runner.

4. Why is `actions/checkout` almost always the first step in a job?
Reveal answer

Because the runner starts as a completely empty, fresh virtual machine. If you don't checkout (download) your code first, none of your subsequent steps will have anything to build or test.
