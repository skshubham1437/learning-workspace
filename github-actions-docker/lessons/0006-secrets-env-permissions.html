Lesson 06

Secrets, Environment Variables & Permissions

Securely pass API keys to your workflow and grant the exact permissions it needs.

📖 Primary source: Using secrets in GitHub Actions and Variables.

The Problem

Real-world pipelines often need to talk to the outside world. Maybe your tests need to query a staging database, or your deployment job needs an AWS access key, or your linter needs an API token.

You must never hardcode these values in your repository. If you hardcode a secret in your workflow YAML, anyone with read access to the repo can steal it. And even for public repositories, GitHub will often automatically revoke leaked tokens.

Environment Variables (env)

First, let's talk about non-secret values. You can define environment variables in your workflow using the env: key. These are available to the steps running in your job.

You can define env: at three levels:

  • Workflow level: Available to all jobs and steps.
  • Job level: Available to all steps in that job.
  • Step level: Available only to that specific step.
env:
  NODE_ENV: development  # Workflow-level env

jobs:
  test:
    runs-on: ubuntu-latest
    env:
      PORT: 8080         # Job-level env
    steps:
      - name: Run tests
        run: npm test
        env:
          LOG_LEVEL: debug # Step-level env

Secrets

For sensitive values like API keys, you use GitHub Secrets. You store these securely in your repository settings (Settings > Secrets and variables > Actions). Once saved, you cannot view the secret again—you can only update or delete it.

To access a secret in your workflow, you use the ${{ secrets.SECRET_NAME }} expression context.

Best Practice

Do not pass secrets directly into a run: script command like npm test ${{ secrets.API_KEY }}. This can lead to injection attacks if the secret contains special characters. Instead, map the secret to an environment variable in the step.

Here is the safe way to use a secret:

steps:
  - name: Run tests that need a DB password
    run: npm test
    env:
      DB_PASSWORD: ${{ secrets.DB_PASSWORD }}

If your script prints the environment variable, GitHub will automatically mask the secret in the workflow logs, replacing it with ***.

The GITHUB_TOKEN & Permissions

Sometimes, your workflow needs to interact with GitHub itself (e.g., adding a comment to a PR, or creating a release). You don't need to create a personal access token for this!

GitHub automatically creates a unique secret named GITHUB_TOKEN for every workflow run. You access it via ${{ secrets.GITHUB_TOKEN }}.

Permissions

By default, the GITHUB_TOKEN has restricted permissions. If you want your workflow to do more—or if you want to follow the principle of least privilege and restrict it further—you use the permissions: block.

permissions:
  contents: read     # Allow checking out the code
  pull-requests: write # Allow commenting on PRs

You can define permissions: at the top level of the workflow or at the job level. If you define it, all other permissions are set to "none".

🧠 Retrieval Practice

1. What is the safest way to pass a secret to a command in a step?
Reveal answer

By mapping it to an environment variable using the env: block, e.g., env: API_KEY: ${{ secrets.API_KEY }}. Passing it directly into the run: script can lead to script injection.

2. Can you retrieve the value of a secret from the GitHub UI after saving it?
Reveal answer

No. Once saved, the secret is encrypted and cannot be viewed. It can only be updated, deleted, or used by workflows.

3. Why should you define a permissions: block when using GITHUB_TOKEN?
Reveal answer

To follow the principle of least privilege. By explicitly defining what the token can do (e.g., contents: read), you prevent accidental or malicious modifications to your repository if a step is compromised.
