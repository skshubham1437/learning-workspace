Lesson 10

Docker Networks & Container Communication

How containers find and talk to each other — and why localhost is not the answer.

📖 Primary source: Docker Networking overview · Bridge network driver

The Core Problem: Containers Are Islands

Every Docker container lives inside its own isolated network namespace. Two containers running side-by-side on the same machine are, by default, completely invisible to one another.

If you run a Node.js API in container A and a PostgreSQL database in container B, the API cannot connect to the database using localhost. Inside container A, localhost resolves to container A's own loopback interface—not to container B. The database doesn't exist in container A's universe.

The Kubernetes parallel

In Kubernetes, this maps directly to the concept of Services. A K8s Service gives a stable DNS name to a group of pods. Without it, pods are also islands that can't find each other. Learning Docker networks now makes Kubernetes Services feel like a natural extension—not a new concept.

Docker Network Types

Docker ships with several network drivers. You'll encounter three in practice:

  • bridge (default): A private internal network on your host. Containers on the same bridge network can communicate. Containers on different bridge networks cannot. This is what you'll use for almost everything.
  • host: Removes network isolation entirely. The container shares your host machine's network stack directly. Port 3000 in the container is port 3000 on your Mac—no mapping needed. Only available on Linux (not Mac or Windows Docker Desktop).
  • none: Complete network isolation. Container has no network interface at all. Used for batch jobs that shouldn't touch the network.

User-Defined Bridge Networks: The Right Way

Docker creates a built-in bridge network called bridge (also known as docker0). You can connect containers to it, and they'll be able to reach each other by IP—but not by name. The default bridge network does not provide DNS.

User-defined bridge networks solve this. When you create your own network, Docker automatically runs a DNS server for it. Every container on that network is reachable from any other container by its --name.

# Create your own private network
docker network create my_app_net

# Start a postgres database on that network
# The container is named 'postgres_db' — this becomes its DNS hostname
docker run -d \
  --name postgres_db \
  --network my_app_net \
  -e POSTGRES_PASSWORD=secret \
  -e POSTGRES_DB=myapp \
  postgres:15

# Start an API container on the SAME network
# The connection string uses 'postgres_db' as the hostname — not 'localhost'
docker run -d \
  --name api_server \
  --network my_app_net \
  -p 3000:3000 \
  -e DATABASE_URL=postgresql://postgres:secret@postgres_db:5432/myapp \
  my_api_image

Notice what just happened: the API container uses postgres_db as the hostname in its database URL. Docker resolves that name to the correct container's IP address automatically.

The Expose vs Publish Distinction

This is one of the most misunderstood things in Docker. There are two completely separate mechanisms for ports, and conflating them causes real security problems.

Mechanism Flag Who can access?
Automatic (network) none needed Other containers on the same Docker network. All ports are open between them.
EXPOSE in Dockerfile EXPOSE 5432 Nobody. It is documentation only. It does not open any port to anyone.
Publish -p 5432:5432 Your host machine and, depending on firewall rules, the public internet.

The security implication: never publish your database port (-p 5432:5432) in production. Your API container can reach the database over the Docker network without it. Publishing a database port exposes it to the internet.

Inspecting a Network

When things go wrong—a container can't reach another—your first tool is docker network inspect. It shows you exactly which containers are connected and their internal IPs.

docker network inspect my_app_net

Look for the "Containers" key in the output. If your container isn't listed there, it's not on the network—that's your bug.

Your Turn: Prove DNS Works

Run this experiment yourself. You'll prove that two containers can discover each other by name on a user-defined network—something that is impossible on the default bridge.

  1. Create a network:
    docker network create test_net
  2. Start a "database" container (just a sleeping Alpine—no real DB needed):
    docker run -d --name my_database --network test_net alpine sleep 600
  3. Start a second "API" container on the same network and drop into its shell:
    docker run -it --rm --name my_api --network test_net alpine sh
  4. From inside the API container, ping the database by its container name:
    ping my_database
    You should see packets responding. Press Ctrl+C to stop.
  5. Now try to reach it by IP. Run nslookup my_database inside the shell to see what IP Docker assigned it. Note the IP, then ping [that IP] directly—it also works.
  6. Exit: exit
  7. Now repeat the experiment without a custom network. Run the database on the default bridge (omit --network test_net), start a new API shell, and try ping my_database. You'll see it fail—name resolution doesn't exist on the default bridge.
  8. Clean up: docker rm -f my_database && docker network rm test_net

The fact that name resolution works on user-defined networks but not the default bridge is the reason every real Docker setup always creates a named network—whether you write it manually or use Docker Compose.
