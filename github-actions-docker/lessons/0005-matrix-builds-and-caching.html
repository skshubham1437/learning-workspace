Lesson 05

Matrix Builds & Caching

Test across multiple Node.js versions in parallel — and make npm ci fast by caching your dependencies.

📖 Primary source: Running variations of jobs in a workflow and Dependency caching — GitHub's official docs on both features.

The Problem

In Lesson 04, our CI pipeline tested on one version of Node.js: 20. That works — but in the real world you often need to answer: does my code work on Node 18, 20, and 22?

You could copy-paste the job three times, changing the version each time. But that's fragile and ugly. GitHub Actions has a first-class solution: the strategy matrix.

What a Matrix Does

A matrix tells GitHub: "run this same job once for each combination of values I give you." You define variables (like node-version) with a list of values, and Actions automatically creates one job per value — running them in parallel by default.

Mental Model

Think of a matrix as a for loop over your job. You write the job once, the matrix iterates over it with different parameters. Each iteration is a completely independent job on its own runner.

Here's the Lesson 04 test job, upgraded with a matrix:

test:
  needs: lint
  runs-on: ubuntu-latest

  strategy:
    matrix:
      node-version: [18, 20, 22]

  steps:
    - name: Checkout code
      uses: actions/checkout@v4

    - name: Setup Node.js ${{ matrix.node-version }}
      uses: actions/setup-node@v4
      with:
        node-version: ${{ matrix.node-version }}

    - name: Install dependencies
      run: npm ci

    - name: Run tests
      run: npm test

That single job definition produces three parallel jobs: one with Node 18, one with 20, one with 22. GitHub labels them test (18), test (20), test (22) in the Actions UI.

Anatomy of a Matrix

Let's unpack the syntax piece by piece:

KeyWhat It Does
strategy: A job-level block that configures how the job is run. The matrix lives inside it.
matrix: Defines the variables and their values. Each variable becomes accessible via ${{ matrix.<name> }}.
node-version: [18, 20, 22] A custom variable name with three values. GitHub spins up one job per value.
Analogy

Imagine you're a restaurant with three kitchens. Instead of writing three separate recipes (one per kitchen), you write one recipe and say: "Kitchen A uses ingredient X, Kitchen B uses ingredient Y, Kitchen C uses ingredient Z." Same recipe, different inputs, all cooking simultaneously.

Accessing matrix values

Inside any step, you reference a matrix variable with the expression syntax: ${{ matrix.node-version }}. GitHub replaces this with the actual value for each job instance.

This is why the name: field says Setup Node.js ${{ matrix.node-version }} — the logs will show "Setup Node.js 18", "Setup Node.js 20", etc., making it easy to identify which job is which.

Key Concept — Expressions

The ${{ }} syntax is GitHub Actions' expression language. It can access contexts — objects like matrix, github, env, secrets, and more. For now, just know that matrix.* gives you the current iteration's value. We'll use more contexts in Lesson 06 when we cover secrets.

Multi-Dimensional Matrices

You can add more than one variable. Every combination runs:

strategy:
  matrix:
    node-version: [18, 20]
    os: [ubuntu-latest, windows-latest]

This creates 4 jobs (2 × 2): Node 18 on Ubuntu, Node 18 on Windows, Node 20 on Ubuntu, Node 20 on Windows. You'd then use ${{ matrix.os }} in runs-on:.

For our mission (React, Node, Go apps), we'll mostly use single-dimension matrices to test across runtime versions. Multi-dimensional is nice to know — you'll reach for it when you need cross-platform testing.

Controlling Matrix Behaviour

Two important options live under strategy::

fail-fast

strategy:
  fail-fast: false
  matrix:
    node-version: [18, 20, 22]

By default, fail-fast is true: if any matrix job fails, GitHub cancels all the other running jobs. This saves minutes — but sometimes you want to see all failures at once (e.g., "does it break on Node 18 and 22, or just 18?"). Setting fail-fast: false lets every job run to completion regardless.

When to use each

fail-fast: true (default) — fast feedback on PRs. One failure means "fix this first."
fail-fast: false — debugging compatibility issues. You need the full picture across all versions.

max-parallel

strategy:
  max-parallel: 2
  matrix:
    node-version: [18, 20, 22]

Limits how many matrix jobs run simultaneously. Useful if you're on a GitHub Free plan (20 concurrent jobs) and want to leave room for other workflows. Usually you don't need this.

The Speed Problem

We now have 3 test jobs (one per Node version) plus the lint job — that's 4 jobs total, each doing npm ci from scratch. On a project with many dependencies, npm ci can take 30–60 seconds. Multiply that by 4 jobs, and you're wasting a lot of time downloading the same packages over and over.

This is where dependency caching comes in.

How Caching Works

GitHub Actions can save a directory (like ~/.npm, npm's download cache) to its servers after a workflow run. On the next run, if the cache key matches, it restores the directory instead of downloading everything again.

Important Distinction

Caching saves the download cache (~/.npm), not node_modules/ itself. npm ci still runs and creates node_modules/ from the cache — but it's dramatically faster because it doesn't need to fetch packages from the internet. This is the recommended approach per GitHub's caching docs.

The cache key

A cache key is a string that uniquely identifies what's cached. For npm, the standard pattern is:

key: ${{ runner.os }}-node-${{ hashFiles('**/package-lock.json') }}

Breaking that down:

  • runner.os — the operating system (e.g., Linux). Different OSes need different caches.
  • hashFiles('**/package-lock.json') — a hash of your lockfile. When dependencies change, the lockfile changes, the hash changes, and the cache is invalidated. Perfect.

The Easy Way: Built-In Caching

Good news: actions/setup-node@v4 has built-in caching support. You don't need a separate caching step. Just add one line:

- name: Setup Node.js ${{ matrix.node-version }}
  uses: actions/setup-node@v4
  with:
    node-version: ${{ matrix.node-version }}
    cache: 'npm'              # ← this one line

That's it. Under the hood, actions/setup-node uses actions/cache to save and restore ~/.npm, keyed on your package-lock.json hash. First run: no cache, normal speed. Second run: cache hit, npm ci runs in seconds instead of 30–60 seconds.

Analogy

Imagine a kitchen that orders ingredients from a supplier every time it opens (that's the fresh runner). Caching is like having a pantry that remembers what you ordered last time. If the recipe hasn't changed (lockfile hash matches), the pantry already has everything — you skip the delivery wait.

Important Prerequisite

The cache: 'npm' option requires a package-lock.json in your repository. This is why we ran npm install locally in Lesson 04 — it generates the lockfile that both npm ci and the cache mechanism depend on.

The Full Workflow

Here's our Lesson 04 workflow upgraded with matrix builds and caching:

name: "CI"

on:
  push:
    branches: [ "main" ]
  pull_request:
    branches: [ "main" ]

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'             # ← cached!

      - name: Install dependencies
        run: npm ci

      - name: Run linter
        run: npm run lint

  test:
    needs: lint
    runs-on: ubuntu-latest

    strategy:                        # ← NEW: matrix strategy
      matrix:
        node-version: [18, 20, 22]

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Setup Node.js ${{ matrix.node-version }}
        uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node-version }}
          cache: 'npm'             # ← cached!

      - name: Install dependencies
        run: npm ci

      - name: Run tests
        run: npm test

What changed from Lesson 04:

  1. strategy.matrix added to the test job — now runs on Node 18, 20, and 22 in parallel.
  2. cache: 'npm' added to both actions/setup-node steps — second run onwards will be much faster.
  3. ${{ matrix.node-version }} replaces the hardcoded '20' in the test job's node-version field.

How the Jobs Look in the Actions UI

When this workflow runs, the Actions tab shows a dependency graph like this:

  ┌──────────┐
  │   lint   │
  └────┬─────┘
       │ needs
  ┌────┴─────────────────────────────┐
  │                                  │
┌─┴──────────┐ ┌────────────┐ ┌─────┴──────┐
│ test (18)  │ │ test (20)  │ │ test (22)  │
└────────────┘ └────────────┘ └────────────┘

Lint runs first. Once it passes, all three test jobs launch simultaneously. If any test job fails and fail-fast is true (the default), the others are cancelled. The whole workflow takes roughly as long as lint + the slowest test job — not lint + three sequential tests.

The Manual Way: actions/cache

The built-in cache: 'npm' on actions/setup-node handles npm automatically. But other ecosystems (Go modules, pip, etc.) don't have this convenience. For those, you use actions/cache directly:

- name: Cache npm dependencies
  uses: actions/cache@v4
  with:
    path: ~/.npm
    key: ${{ runner.os }}-npm-${{ hashFiles('**/package-lock.json') }}
    restore-keys: |
      ${{ runner.os }}-npm-

Three parameters:

  • path: — what directory to cache. For npm, it's ~/.npm.
  • key: — the exact cache key. Cache hit = exact match.
  • restore-keys: — fallback prefixes. If the exact key isn't found, GitHub looks for keys starting with these prefixes and restores the closest match. You still get most of the cache benefit even if the lockfile changed slightly.
When to use which

Use the built-in cache: option on actions/setup-node (or setup-python, etc.) whenever possible — it's simpler and less error-prone. Use actions/cache directly when you need to cache something the setup action doesn't handle, or when you're working with a language that has no setup action with built-in caching.

Cache Limits

A few things to know about GitHub's cache:

  • 10 GB total per repository. Once you hit the limit, the oldest caches are evicted. (Source)
  • Caches expire after 7 days of not being accessed.
  • Caches are scoped by branch. A cache created on a feature branch can fall back to the main branch cache, but not vice versa.
  • Cache does not persist across workflow re-runs if the key changed.

For most projects, these limits are generous. You won't hit them unless you're caching very large directories (like Docker layer caches — we'll get to those later).

🧠 Retrieval Practice

1. What keyword do you add to a job to run it once per value in a list?
Reveal answer

strategy.matrix — you define a variable (like node-version) with a list of values under strategy: matrix: in the job. Each value creates a separate, parallel job instance.

2. A matrix has node-version: [18, 20] and os: [ubuntu-latest, windows-latest]. How many jobs are created?
Reveal answer

4 jobs — one for every combination: (18, ubuntu), (18, windows), (20, ubuntu), (20, windows). The matrix computes the cross-product of all variable values.

3. What does fail-fast: false do, and when would you use it?
Reveal answer

By default (fail-fast: true), if any matrix job fails, all other running matrix jobs are cancelled. Setting fail-fast: false lets every job finish, so you can see the full picture of which versions pass and which fail. Useful when debugging cross-version compatibility.

4. What does cache: 'npm' on actions/setup-node actually cache — node_modules/ or something else?
Reveal answer

It caches ~/.npm — npm's download cache, not node_modules/. npm ci still creates node_modules/, but it reads packages from the local cache instead of downloading them from the internet, which is much faster.

5. What determines whether a cache is "hit" or "miss"?
Reveal answer

The cache key. For npm, it's typically based on the OS and a hash of package-lock.json. If the key matches a previously saved cache, it's a hit. If the lockfile changed (new dependencies), the hash changes, the key doesn't match, and it's a miss — though restore-keys can provide a partial match.

🔍 Spot the Bug

This workflow has one bug. Try to find it.

jobs:
  test:
    runs-on: ${{ matrix.os }}
    strategy:
      matrix:
        os: [ubuntu-latest, windows-latest]
        node-version: [18, 20]
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
      - run: npm ci
      - run: npm test

Answer

Reveal answer

The node-version in setup-node is hardcoded to '20' instead of using ${{ matrix.node-version }}. The matrix defines node-version: [18, 20], but the step ignores it. All 4 jobs will test on Node 20 — the Node 18 dimension is wasted. The fix: change node-version: '20' to node-version: ${{ matrix.node-version }}.
