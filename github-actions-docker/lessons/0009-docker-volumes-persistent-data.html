Lesson 09

Docker Volumes & Persistent Data

Because losing your production database when a container restarts is generally considered "bad practice".

📖 Primary source: Docker Storage: Volumes

The Problem: Ephemeral Containers

By default, containers are ephemeral. Any data created inside the container is written to a temporary read/write layer. When the container stops or is deleted, that data is destroyed forever.

If you run a MySQL database in a container, it will work perfectly—until the container restarts, at which point your entire database vanishes. To solve this, we use Volumes.

Volumes vs Bind Mounts

There are two primary ways to persist data or share it between the host and the container. They both use the -v or --mount flag.

Deconstructing the Volume Syntax (-v)

The -v (or --volume) flag maps a source directory outside the container to a destination directory inside the container. The syntax is always:

-v [source]:[destination]

  • [source] (Left side of the colon): The data location outside the container.
    • For a Named Volume: This is simply the name of the volume (e.g., my_db_data).
    • For a Bind Mount: This is the absolute path to a folder on your host machine (e.g., $(pwd) or /Users/asdasd/project).
  • [destination] (Right side of the colon): The absolute path inside the container where that data will be mounted (e.g., /var/lib/postgresql/data or /app).

⚠️ Note: Any data written by the container to the [destination] path will automatically be persisted in the [source] location.

1. Named Volumes (Best for Databases)

A Named Volume is a folder managed entirely by Docker. You give it a name, and Docker stores it somewhere deep in its own managed file system (like /var/lib/docker/volumes/ on Linux). You don't care exactly where it is; you just trust Docker to keep it safe across container restarts.

# Create a named volume (optional, docker run will create it automatically if it doesn't exist)
docker volume create my_db_data

# Mount the named volume to where MySQL stores its data internally
docker run -d \
  --name my_postgres \
  -e POSTGRES_PASSWORD=secret \
  -v my_db_data:/var/lib/postgresql/data \
  postgres:15

2. Bind Mounts (Best for Local Development)

A Bind Mount maps a specific folder on your host machine (e.g., your project folder) directly into the container. This is crucial for local development because if you edit your source code on your Mac, the changes instantly appear inside the container without rebuilding the image.

# Mount the current directory (pwd) into the container's /app directory
docker run -d \
  --name dev_server \
  -p 3000:3000 \
  -v $(pwd):/app \
  node:20-alpine npm run dev
Kubernetes Connection

In Kubernetes, this concept translates to PersistentVolumes (PV) and PersistentVolumeClaims (PVC). Kubernetes pods are completely ephemeral. To save state in K8s, your Pod has to "claim" a PersistentVolume to mount inside it. Understanding Docker volumes now makes K8s PVCs much easier to grasp later!

Cleaning Up Volumes

Because volumes persist even when containers are deleted, they can eventually consume a lot of disk space if left unchecked.

# List all volumes
docker volume ls

# Delete a specific volume
docker volume rm my_db_data

# CAUTION: Delete ALL volumes not currently attached to a container
docker volume prune

Your Turn

Try spinning up an ephemeral container, killing it, and proving the data is gone. Then do it again with a volume!

  1. Start an Alpine Linux container in interactive mode:
    docker run -it --name test_ephemeral alpine sh
  2. Inside the container, create a file: echo "hello" > /data.txt. Then type exit.
  3. The container stopped. Start it again and connect:
    docker start -i test_ephemeral. Check the file: cat /data.txt. (It's still there because the container was just stopped, not deleted).
  4. Type exit. Now delete the container: docker rm test_ephemeral. The data is now gone forever.
  5. The Volume test: Run a new container, but attach a named volume:
    docker run -it --name test_volume -v my_first_volume:/my_data alpine sh
  6. Inside the container, write to the volume folder:
    echo "Persistent Hello!" > /my_data/hello.txt. Then exit.
  7. Delete the container: docker rm test_volume.
  8. Create a completely new container, but attach that same volume:
    docker run -it --rm -v my_first_volume:/my_data alpine cat /my_data/hello.txt. You should see your text!
