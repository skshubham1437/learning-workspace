Lesson 02

What Docker Actually Is

Demystifying containers, images, and the Dockerfile.

The "Works on My Machine" Problem

In Lesson 1, we learned that Docker solves the "works on my machine" problem by packaging the application and its environment together. But what does that actually mean?

If you build a Node.js app on your Mac, it depends on:

  • The exact version of Node.js installed on your Mac
  • The exact versions of dependencies in your node_modules
  • Underlying C++ libraries on macOS (if any packages compile native bindings)
  • Environment variables you've set locally

If the deployment server runs Ubuntu and has a slightly older version of Node, your app might crash. Docker fixes this by letting you define the entire OS environment in code.

Containers vs. Virtual Machines

To isolate an app, we used to use Virtual Machines (VMs). A VM simulates an entire computer. It has its own kernel (the core of the OS), its own memory management, and its own hard drive. Running 5 VMs means running 5 full operating systems.

Containers are different. They share the host machine's kernel, but isolate the processes, networking, and file system. They are incredibly lightweight. You can start a container in a fraction of a second, whereas a VM takes minutes to boot.

Analogy

A Virtual Machine is like buying a whole separate house for every person. It's fully isolated, but extremely expensive and takes up a lot of space.

A Container is like living in an apartment building. Everyone shares the same foundation and plumbing (the kernel), but you have your own locked door, your own furniture, and your own space.

Images vs. Containers

These two terms are often used interchangeably, but they are fundamentally different:

  • An Image is the blueprint. It is a read-only template that contains the OS, the runtime (like Node or Go), your code, and your dependencies. Once built, an image never changes.
  • A Container is a running instance of an image. If the image is the blueprint, the container is the house built from it. You can start 10 containers from a single image.
Key distinction

You build an image. You run a container.

The Dockerfile: Writing the Recipe

How do we create an image? We write a Dockerfile. It is simply a list of instructions telling Docker how to assemble the image, step by step.

Here is a basic Dockerfile for a Node/Express app:

# 1. Start from a base image (an OS with Node already installed)
FROM node:20-alpine

# 2. Set the working directory inside the container
WORKDIR /app

# 3. Copy only the package files first
COPY package.json package-lock.json ./

# 4. Install dependencies
RUN npm install

# 5. Copy the rest of the application code
COPY . .

# 6. Specify the command to run when the container starts
CMD ["node", "server.js"]

The Core Instructions

  • FROM: The foundation. Always the first line. node:20-alpine means "give me a tiny Linux OS (Alpine) with Node v20 pre-installed."
  • WORKDIR: Like running cd /app. All subsequent commands run in this folder.
  • COPY: Copies files from your laptop into the image.
  • RUN: Executes a command during the build process (e.g., installing packages).
  • CMD: The default command that runs when a container starts up.

Layers and Caching: Why Order Matters

Docker images are built in layers. Every instruction in a Dockerfile creates a new layer on top of the previous one. Docker caches these layers to make future builds incredibly fast.

If you change a line of code in server.js, Docker looks at the Dockerfile top-to-bottom. It will re-use the cached layers for FROM, WORKDIR, COPY package.json, and RUN npm install because those haven't changed. It only rebuilds from the COPY . . line downwards.

Build 1 (Everything new)
Layer 1: FROM node:20
Layer 2: COPY package.json
Layer 3: RUN npm install
Layer 4: COPY . .
Build 2 (Changed server.js)
Layer 1: (Cached)
Layer 2: (Cached)
Layer 3: (Cached)
Layer 4: COPY . . (Rebuilt)
The Golden Rule of Dockerfiles

Always put the things that change least often at the top, and the things that change most often at the bottom. This is why we copy package.json and run npm install before we copy the rest of the source code!

🧠 Retrieval Practice

Try to answer these from memory before revealing the answers.

1. What is the difference between an Image and a Container?
Reveal answer

An image is the read-only blueprint (the recipe). A container is the running instance of that image (the actual running process). You build an image, and you run a container.

2. What is the difference between the RUN instruction and the CMD instruction?
Reveal answer

RUN executes commands during the build process to assemble the image (like npm install). CMD specifies the default command that runs when a container is started from the image (like node server.js).

3. Why do we copy `package.json` and run `npm install` BEFORE copying the rest of the source code?
Reveal answer

Because of Docker layer caching. Source code changes frequently, but dependencies change rarely. By installing dependencies first, Docker can cache the heavy npm install step, making subsequent builds where you only changed code much faster.

4. Why is a container much faster to start than a Virtual Machine?
Reveal answer

A VM has to boot up a full separate operating system and kernel. A container shares the host machine's kernel and only isolates the application process, so it starts almost instantly.
