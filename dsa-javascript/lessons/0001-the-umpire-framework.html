Lesson 01

The UMPIRE Framework

What to do when you look at a LeetCode problem and your mind goes blank.

The Biggest Mistake in Coding Interviews

When most people are handed a coding problem, their instinct is to immediately start writing code. They read the prompt, panic slightly, and start typing a for loop hoping the solution will reveal itself along the way.

The Trap

Jumping straight to code is the fastest way to fail an interview. It leads to spaghetti code, missed edge cases, and awkward silence.

To fix the "I'm bad at problem solving" feeling, we don't start by learning complex tree algorithms. We start by learning a process. When you have a process, you don't need inspiration—you just follow the steps.

The industry standard process is called UMPIRE.

The UMPIRE Method

U
Understand

Goal: Prove you know exactly what the problem is asking before you try to solve it.

  • Read the problem.
  • Ask clarifying questions: "Can the array be empty? Are there negative numbers? Does casing matter?"
  • Create your own simple test case: "So if the input is [1,2,3], the output should be 6?"
M
Match

Goal: Figure out what category of problem this is.

  • Does this look like a problem you've seen before?
  • Do we need to search? Maybe a Hash Map. Do we need to compare ends of an array? Maybe Two Pointers.
  • You don't need the exact solution here, just the "vibe" of the tools you might need.
P
Plan

Goal: Write out the logic in plain English (or pseudocode) without writing actual code.

  • This is the most important step. Explain to the interviewer (or yourself): "First, I will loop through the array. Second, I will check if the number exists in a hash map..."
  • If you get stuck here, that's fine! It's much easier to fix a broken English sentence than broken JavaScript.
I
Implement

Goal: Translate your English plan into JavaScript.

  • Do not do this until the "Plan" is finished.
  • If you did step P correctly, step I should just be typing. There should be zero heavy thinking here.
R
Review

Goal: Catch your own bugs before the interviewer points them out.

  • Pick a simple test case.
  • Read through your code line by line, acting as the computer. Keep track of what your variables equal at every step.
  • Do not just say "looks good." Actually trace the logic.
E
Evaluate

Goal: Analyze how good your solution is.

  • What is the Time Complexity? (How long does it take as the input grows?)
  • What is the Space Complexity? (How much extra memory does it use?)

Why This Works

The Secret

Interviewers don't actually care about the final code as much as you think. They care about how you think.

If you immediately write perfect code in silence, they don't know if you're a genius or if you just memorized it. If you follow UMPIRE, even if you never finish the "Implement" step, they know you are a methodical, careful engineer who communicates well.

For the rest of this track, we will never jump straight to the code. We will always Understand and Plan first.

🧠 Retrieval Practice

Try to answer these from memory to cement the framework.

1. What is the most common trap people fall into when given a coding problem?
Reveal answer

Jumping straight to writing code without fully understanding the problem or planning an approach.

2. What does UMPIRE stand for?
Reveal answer

Understand, Match, Plan, Implement, Review, Evaluate.

3. What is the goal of the "Understand" phase?
Reveal answer

To clarify the prompt, ask questions about edge cases (like empty arrays or negative numbers), and create simple examples to verify the expected input/output.

4. Why is the "Plan" step so important before moving to "Implement"?
Reveal answer

Because it's much easier to fix broken logic in plain English than it is to untangle broken JavaScript. If the plan is solid, the implementation is just typing.
