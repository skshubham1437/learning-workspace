Lesson 03

JavaScript Arrays and Strings: The 80/20 Interview Guide

Master the essential built-in methods and avoid the hidden O(n²) performance traps that sink technical interviews.

You Don't Need to Memorize Everything

JavaScript has over 40 built-in array methods and dozens of string methods. Trying to memorize all of them before a coding interview is a waste of working memory.

In real-world coding interviews, 80% of problems rely on just a handful of core methods. More importantly, interviewers don't just care if you know a method—they care if you know its Big O time and space complexity.

1. Adding & Removing: The Front vs. The Back

The most important performance distinction in JavaScript arrays is where you insert or remove items.

✓
The Back: push() and pop() — O(1) Constant Time

Adding or removing an element from the end of an array is instantaneous. No other elements are affected.

const stack = [10, 20, 30];
stack.push(40); // [10, 20, 30, 40] -> O(1) time
stack.pop();    // [10, 20, 30]     -> O(1) time
⚠️
The Front: unshift() and shift() — O(n) Linear Time

Adding or removing from the beginning index (0) forces the JavaScript engine to re-index every single remaining item in memory!

const queue = [10, 20, 30];
queue.shift(); // Removes 10, but shifts 20 to index 0, and 30 to index 1 -> O(n) time!
Interview Trap

Never use array.shift() or array.unshift() inside a loop! A loop that runs n times calling an O(n) shift operation degrades your algorithm into sluggish O(n²) quadratic time.

2. Slicing, Splitting, and Joining

When manipulating strings and arrays, be mindful of when you are creating new copies in memory (Space Complexity):

  • arr.slice(start, end) — Returns a shallow copy of a portion of an array. Time: O(k), Space: O(k) (where k is the size of the slice).
  • arr.splice(start, deleteCount, ...items) — Modifies an array in-place by removing or replacing elements. Time: O(n), Space: O(1).
  • str.split('') & arr.join('') — Converts strings to arrays and vice versa. Time: O(n), Space: O(n).

3. The Hidden O(n²) Trap: indexOf and includes

Many candidates write code like this and assume it runs in O(n) linear time because there is only one visible for loop:

function findCommonElements(arr1, arr2) {
  const common = [];
  for (let item of arr1) {          // Loop runs O(n) times
    if (arr2.includes(item)) {      // ⚠️ HIDDEN O(m) LINEAR SCAN!
      common.push(item);
    }
  }
  return common;
}
Under the Hood

Methods like .includes(), .indexOf(), and .find() do not magically jump to the answer. They perform a linear scan from index 0 to the end. Calling them inside a loop multiplies your complexity: O(n × m) (or O(n²) if arrays are equal size). In Phase 3, we will learn how to fix this using Hash Sets!

Interactive Retrieval Practice

Test Your Understanding

Why is calling array.shift() inside a while loop considered a dangerous performance trap in technical interviews?

Correct! Removing from index 0 forces every subsequent item at index 1, 2, 3... to shift left in memory. Doing an O(n) re-indexing operation inside an n-iteration loop results in O(n²) quadratic time complexity.

Analyze the code below. What is its true worst-case time complexity?

function containsVowel(words) {
  const vowels = ['a', 'e', 'i', 'o', 'u'];
  for (let word of words) {
    if (vowels.includes(word[0])) { return true; }
  }
  return false;
}
Correct! This is a tricky distinction! Even though vowels.includes() is a linear scan inside a loop, the vowels array has a fixed constant size of 5. Scanning 5 items is O(5), which simplifies to O(1) constant time! Thus, the overall loop running over n words is simply O(n) linear time!

Interactive Coding Challenge: The O(n) Palindrome

Practice using index lookups without creating a reverse copy in memory (which would cost O(n) space). Write a function that returns true if a string is a palindrome, using only O(1) constant auxiliary space!

Reveal Solution & Explanation 
function isPalindrome(str) {
  let left = 0;
  let right = str.length - 1;
  
  while (left < right) {
    if (str[left] !== str[right]) return false;
    left++;
    right--;
  }
  return true;
}

Explanation: By using two pointers (one at the beginning and one at the end), we can check if characters match while moving inwards. This completely avoids allocating a new reversed string in memory, keeping our space complexity at O(1) constant space.
