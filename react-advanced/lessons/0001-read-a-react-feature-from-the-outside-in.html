Lesson 01

Read a React Feature From the Outside In

Your first production-reading skill: trace what renders, what changes, and who owns the change.

React apps become readable when you stop trying to understand every line at once. The official React docs recommend breaking UI into a component hierarchy, finding the minimal changing state, and placing that state where all dependent components can reach it. That is our first code-reading tool.

For now, use this small feature as a stand-in for the missing `xley-frontend` path. The same reading move will work when we inspect the real project. 

function OrdersPage({ orders }) {
  const [searchText, setSearchText] = useState("");
  const [status, setStatus] = useState("all");

  const visibleOrders = orders.filter((order) => {
    const matchesText = order.customer.includes(searchText);
    const matchesStatus = status === "all" || order.status === status;
    return matchesText && matchesStatus;
  });

  return (
    <>
      <OrderFilters
        searchText={searchText}
        status={status}
        onSearchTextChange={setSearchText}
        onStatusChange={setStatus}
      />
      <OrderList orders={visibleOrders} />
    </>
  );
}

The outside-in pass

1. What renders?

`OrdersPage` renders `OrderFilters` and `OrderList`. That gives you the feature boundary.

2. What changes?

`searchText` and `status` change over time. `visibleOrders` is recalculated from them.

3. Who owns it?

`OrdersPage` owns the filter state because both filters and list need it.

The important distinction: `visibleOrders` feels like data, but it is not state. It is derived during render from `orders`, `searchText`, and `status`. React's docs call this the search for the minimal representation of UI state.

Why this matters

Beginners often add state for anything that appears on screen. Production React code asks a sharper question: can this value be computed from existing props or state? If yes, compute it during render first. If no, store it in state, a reducer, context, the URL, a server-state cache, or a global store.

Later, this same question decides whether you need a package. API data often belongs in TanStack Query. Complex app-wide client state may belong in Redux Toolkit. But filtered arrays like `visibleOrders` usually do not need either.

Your five-minute drill

  1. Open any React page component you already understand a little.
  2. Write down its immediate child components.
  3. Circle every `useState`, reducer, query hook, router hook, and custom hook.
  4. Pick one displayed value and trace whether it is original state or derived data.
  5. Ask me about the first confusing value you find.
